JVM 学习笔记 - 堆

学习笔记 java

堆 - Heap

  • 每一个 JVM 实例只存在一个堆空间，堆也是 Java 内存管理的核心区域

  • Java 堆在 JVM 启动时创建，其大小也就确定了。

    1-Xmx16g # 最大堆内存 16G = -XX:MaxHeapSize
2-Xms8g  # 起始堆内存 8G = -XX:InitialHeapSize
3-XX:+PrintGCDetails # 打印GC的详细信息
    • 一般会把 -Xms 和 -Xmx 的值配置相同的值，为了在垃圾回收机制清理完堆区后不需要重新分隔计算堆区的大小，从而提高性能

    • 默认大小

      -Xms = 物理内存 / 64

      -Xmx = 物理内存 / 4

  • Java 堆空间物理上可以不连续，逻辑上必须连续。

  • 所有线程共享 Java 堆空间，但是堆中有线程私有得缓冲区（TLAB - Thread Local Allocation Buffer）

  • 规范：所有得对象实例和数据都应当在运行时分配在堆上。

    实际使用中，“几乎”所有的对象实例都在堆上分配

  • 栈得栈帧中保存引用，引用指向对象或数据在堆中得位置。

  • 方法结束后，堆中得对象不会马上被移除，在垃圾回收时被移除。

  • 堆是 GC 执行得重点区域

堆空间内部结构

JDK7
  • 新生区 - Young Generation Space
    • Eden
    • Survivor
  • 老年区 - Tenure Generation Space
  • 永久区 - Permanent Space

JDK8
  • 新生区 - Young Generation Space
    • Eden
    • Survivor
  • 老年区 - Tenure Generation Space
  • 元空间 - Meta Space

新生代与老年代

  • 新生代 + 老年代 = -Xmx

  • 新生代与老年代在堆中的空间占比

    1-XX:NewRatio=2 #默认情况 新生代:老年代 = 1:2

  • 新生代空间结构占比

    1-XX:SurvivorRatio=8 #调整 Eden 和 Survivor 的空间占比，默认 8:1:1
2-XX:-UseAdaptiveSizePolicy # 关闭自适应内存分配策略

  • 新生代空间大小

    1-Xmx #设置新生代空间大小（会覆盖 -XX:NewRatio 的配置）

对象分配过程

Minor GC、Major GC 、Full GC

针对 HotSpot VM 的实现，GC 按照回收区域分为部分收集（Partial GC）、整堆收集（Full GC）

  • 部分收集

    • 新生代收集：Minor GC /Young GC，只是新生代的垃圾收集

    • 老年代收集：Major GC/Old GC，只是老年代的垃圾收集

      目前只有 CMS GC 会有单独的收集老年代的行为

      很多时候 Major GC 会和 Full GC 混淆使用

    • 混合收集：Mixed GC，收集整个新生代以及部分老年代的垃圾收集。

      G1 GC

  • 整堆收集：收集整个 Java 堆和方法区的垃圾收集

Minor GC 触发机制

当年轻代空间不足时（Eden 区域满），触发 Minnor GC。Survivor 区域满并不会触发 Minor GC。但是每次 Minor GC 会清理 Eden 和 Survivor 区域。

  • 触发频繁
  • 回收速度快
  • 引发 STW
Major GC 触发机制

Major GC 经常会伴随至少一次 Minor GC，也就是在老年代空间不足时，会先尝试触发 Minor GC。如果之后空间还不足，则触发 Major GC。

Parallel Scavenge 收集器的收集策略有直接进行 Major GC 的策略选择

  • 速度比 Minor GC 满 10 倍以上，STW 时间更长。
  • Major GC 后内存还不足，会报 OOM
Full GC 触发机制

Full GC 时开发或调优中尽量要避免的

  1. 调用 system.gc() 时，系统建议执行 Full GC，但是不是必然执行。
  2. 老年代空间不足
  3. 方法区空间不足
  4. 通过 Minor GC 后进入老年代的平均大小大于老年代的可以用内存
  5. 由 Eden 区、From Space 向 To Space 区复制时，对象大小大于 To Space 的可用内存，则把该对象转存到老年代，且老年代的可用内存小于该对象大小

堆空间分代思想

为什么需要分代？不分代能否正常进行垃圾收集？

分区/分代实际上是把对象分开处理，尽量减少 GC 时对目标对象的检索时间。

内存分配策略

对象提升规则

  • 优先分配 Eden

  • 大对象直接分配到老年代

  • 长期存活的对象分配到老年代

  • 动态年龄判断

    如果 survivor 区中，某个年龄的所有对象大小总和大于 Survivor 空间的一半，则大于等于该年龄值的所有对象直接进入老年代。

    每次 Minor GC，Survivor 区都会进行对象复制，如果一个年龄段的对象占据空间大小超过 Survivor 空间的一半，那么将其移入老年代后可以提高 Minor GC 的效率

  • 空间分配担保

    Survivor 区放不下的对象，由老年代进行空间分配担保，直接放入老年代

    1-XX:HandlePromotionFailure

TLAB - Thread Local Allocation Buffer

why

堆区是线程共享的，由于对象的创建是十分频繁的，所以在并发环境下从堆区划分内存空间是不安全的。为了避免多线程操作统一地址时的加锁的机制，通过 TLAB 保证内存分配的线程安全和分配速度。

what

对 Eden 区，为每个线程分配一个私有的缓存区域（TLAB）。

在多线程同时分配内存时，通过 TLAB 可以保证线程安全并且提升内存分配的吞吐量，这种内存分配方式被称之为 快速分配策略。

  • JVM 优先在 TLAB 上进行内存分配。但是不是所有的对象都能够在 TLAB 上分配成功。

  • 相关配置参数

    1-XX:UserTLAB #开启 TLAB，默认开启
2-XX:TLABWasteTargePercent #设置 TLAB 空间所占 Eden 空间的百分比，默认为 1%

  • 一旦对象在 TLAB 空间分配内存失败，JVM 会尝

堆是分配对象存储的唯一选择么？

逃逸分析 - Escape Analysis

通过逃逸分析，Java Hotspot 编译器能够分析出一个新的对象的引用的使用范围，从而决定是否需要将这个对象分配到堆上。

逃逸分析的基本型ing 为就是分析对象动态作用域：

  • 当一个对象在方法中被定义后，对象只能在方法内部使用，则认为没有发生逃逸
  • 当一个对象在方法中被定义后，它被外部方法所引用，则认为发生了逃逸。
 1/**
 2 * 逃逸分析实例代码
 3 */
 4public class EscapeAnalysisTest {
 5    private InnerObj innerObj;
 6    
 7    static class InnerObj {
 8    
 9    }
10    
11    /**
12     * 发生逃逸
13     */
14    public InnerObj getInnerObj() {
15        return innerObj == null ? new InnerObj() : innerObj;
16    }
17    
18    /**
19     * 发生逃逸
20     */
21    public void setInnerObj() {
22        this.innerObj = new InnerObj();
23    }
24    
25    /**
26     * 未发生逃逸
27     */
28    public void method_01() {
29        InnerObj a = new InnerObj();
30    }
31    
32    /**
33     * 发生逃逸
34     */
35    public void method_02() {
36        // method_02 调用了 getInnerObj 方法，对象在 getInnerObj 方法结束后不能被回收，所以发生了逃逸
37        InnerObj a = getInnerObj();
38    }
39}

同步省略

如果一个对象被发现只能从一个线程被访问到，那么对于这个对象的操作可以不考虑同步。

线程私有的对象相关的操作可以省略掉同步操作（例如加锁操作）。

1public class 同步省略示例 {
2    public static void main(String[] args) {
3        Object a = new Object();
4        synchronized (a) {
5            System.out.println("进入同步代码块");
6        }
7    }
8}

标量替换（分离对象）

标量是指一个无法再被分解成更小的数据的数据。还可以被分解的数据叫聚合量。Java 对象就是聚合量

在 JIT 阶段，如果经过逃逸分析，某个对象不会被外界访问，那么经过 JIT 优化，就会把这个对象拆解成若干个 其中包含的若干成员变量 来替代。这个过程就是 变量替换。

 1public class 标量替换示例 {
 2    public static void main(String[] args) throws InterruptedException {
 3        long start = System.currentTimeMillis();
 4        
 5        for (int i = 0; i < 1000 * 10000; i++) {
 6            allocation();
 7        }
 8        
 9        System.out.println("cast time: " + (System.currentTimeMillis() - start) + "豪秒");
10    }
11    
12    private static void allocation() {
13        Person a = new Person();
14        a.age = 18;
15        a.name = "小红";
16    }
17}

标量替换需要在 Server 模式下开启，64 位 jdk 默认开启 Server 模式

1-server # 开启Server模式 64位默认开启
2-XX:+DoEscapeAnalysis #开启逃逸分析 默认开启
3-XX:+EliminateAllocations #开启标量替换

Hotspot VM 并未实现栈上分配，但是应用了标量替换，对于 Hotspot VM 来说，所有的对象都是在堆上分配的。

JDk 6 及其之前，静态变量分配在永久代上。

JDK 7，有永久代，字符串常量池、静态变量分配在堆上。

JDK 8 及其之后，元空间取代了永久代，类型信息、字段、方法、常量在元空间中分配，字符串常量池和静态变量依旧在堆上分配。
