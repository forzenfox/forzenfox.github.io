Flyway - 数据库版本控制工具

2023-06-05
   
数据库 flyway 工具
    0 0

简介

一个能对数据库变更做版本控制的工具
Flyway 官方文档

为什么要使用 Flyway

实际的项目开发中,数据库的变更也需要版本控制,以便解决如下问题:

  1. 写的 sql 忘了是否在所有环境都执行过了,需要一个个切换环境数据库去确认
  2. sql 脚本修改过忘记归档,导致执行遗漏,测试环境出现问题甚至生产环境出现问题
  3. 每次上线 需要人为的执行 Sql 脚本,稳定性不可控.

Flyway 运作方式

官方文档

Flyway 命令

Migrate

官方描述:

Migrates the schema to the latest version. Flyway will create the schema history table automatically if it doesn’t exist.

迁移指定的数据库 schema 到最新的版本.如果没有初始化过,会新建 flyway_schema_history 表.

eg : 最新的版本为 202104290010,数据库的版本为 202104290005
Migrate 命令会依次执行 202104290005->202104290010 的所有版本

eg : 数据库版本与最新版本相同
Migrate 命令不会有任何操作

Clean

官方描述:

Drops all objects in the configured schemas.

Clean 命令会清空配置指定的 schemas 下面的所有对象和数据! 记住是所有,也就说不受 Flyway 版本控制的对象也会被清除掉.

do not use against your production DB!
生产环境应该禁用该命令,测试环境应该谨慎使用

Info

官方描述:

Prints the details and status information about all the migrations.

查看 Flyway 的迁移信息,例如:哪些迁移成功了,哪些版本执行了,执行状态等信息

Validate

官方描述:

Validates the applied migrations against the available ones.

通过 flyway_schema_history 的 checksum 字段校验本地迁移脚本是否和已经执行的迁移脚本一致.

自定义校验规则

Undo

官方描述:

Undoes the most recently applied versioned migration.

简单的说,版本回退.

付费版 Flyway 的功能,没有实践过

Baseline

官方描述:

Baselines an existing database, excluding all migrations up to and including baselineVersion.

简单的理解,用来初始化不为空的数据库.比如:现在有一个项目中途使用 Flyway 进行数据库的版本管理,但是之前很多表已经创建了,不可能清空数据库,这时候可以通过该命令基于数据的现状创建一个基线版本,以后就基于该基线版本进行版本控制.

重置基线

Repair

官方描述:

Repairs the schema history table.

主要针对 flyway_schema_history 表进行修复,主要的用法:

  1. 删除迁移失败的记录(仅针对不支持 DDL 事务的数据库)
  2. 根据迁移脚本修改记录表的信息
  3. 标记所有丢失的迁移记录为删除

24
25    @Bean

28        Assert.isTrue(property.verifyRequired(), "flyway 必填参数为空!");
29
30        FluentConfiguration configuration = Flyway.configure()
31                .dataSource(property.getUrl(), property.getUser(), property.getPassword())
32                .locations(property.getLocations().stream().filter(StringUtils::isNotBlank).toArray(String[]::new))
33                .baselineOnMigrate(property.isBaselineOnMigrate())
34                .validateOnMigrate(property.isValidateOnMigrate())
35                .cleanDisabled(property.isCleanDisabled())
36                .cleanOnValidationError(property.isCleanOnValidationError());
37
38        String[] schemas = property.getSchemas().stream().filter(StringUtils::isNotBlank).toArray(String[]::new);
39        if (schemas.length > 0) {
40            configuration.schemas(schemas);
41        }
42        if (StringUtils.isNotBlank(property.getTable())) {
43            configuration.table(property.getTable());
44        }
45        if (StringUtils.isNotBlank(property.getBaselineVersion())) {
46            configuration.baselineVersion(property.getBaselineVersion());
47        }
48
49        Flyway flyway = configuration.load();

3. 哪种场景会用到多个 schema?

flyway 的多 schema,一般我们都用不到(一般情况我们一个应用只访问一个库),并且如果在数据库访问 url 中指定了数据库的话,schema 可以为空。但是还是会存在某些特殊的应用可能需要访问多个库，比如上述的 spring 集成样例中，我使用了 mysql 的两个不用的库：test 和 flyway_test。

关于 org.flywaydb.core.api.configuration.FluentConfiguration#schemas 官方的注释如下：

Sets the schemas managed by Flyway. These schema names are case-sensitive. (default: The default schema for the database connection)
Consequences:
Flyway will automatically attempt to create all these schemas, unless the first one already exists.
The first schema in the list will be automatically set as the default one during the migration.
The first schema in the list will also be the one containing the schema history table.
The schemas will be cleaned in the order of this list.
If Flyway created them, the schemas themselves will as be dropped when cleaning.

大概意思如下：

1Flyway 管理的schema。 这些schema名称区分大小写。
21. Flyway 将自动尝试创建所有schema，当第一个schema已经存在时，不会去创建后续的scheam。
32. 列表中的第一个schema将在迁移期间自动设置为默认schema。
43. 列表中的第一个schema也将是迁移历史记录表所在的schema。
54. 执行清理命令时,按该顺序删除schema。
65. 如果schema是由Flyway创建的，schema将在执行清理命令时被删除。

以 spring 集成的配置举例，flyway.dataSource.schemas=flyway_test,test,所以 flyway 只会在 flyway_test 中创建迁移历史记录表，并且如果执行的 sql 中不指定 schema 的话，都会默认在 flyway_test 中执行。

1create table tb_person
2(
3    id   int          not null,
4    name varchar(100) not null
5);

1create table test.tb_test
2(
3    id   int          not null,
4    name varchar(100) not null
5);

上例中，tb_person 会创建在 flyway_test 中，tb_test 会创建在 test 中。

4. 历史项目如何使用 flyway？

我们经常会遇到对历史项目集成 flyway 的情况。之前的历史脚本想用 flyway 归档起来，然后基于某个版本之后的脚本，用使用 flyway 去自动管理。

❤️ 举个栗子：❤️

现有一个项目，数据库中已经创建了表：tb_01、tb_02、tb_03。

整改的得到历史脚本如下：V20210425160846__add_table_tb_01.sql、V20210425170846__add_table_tb_02.sql、V20210425180846__add_table_tb_03.sql。

现有新需求，需要新增一个表：tb_04，新增脚本 V20211109160846__add_table_tb_04.sql。

修改配置文件如下：

1flyway.baselineOnMigrate=true
2flyway.baselineVersion=20210425180846
